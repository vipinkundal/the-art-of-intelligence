23. AI Safety, Security, Privacy And Governance

AI safety, security, privacy, and governance are required engineering knowledge, not an optional ethics appendix. Responsible AI work combines robustness, threat modeling, privacy protection, measurement, documentation, accountability, and risk management.

Risk workflow

Map risks Identify model, data, user, system, deployment, security, privacy, and governance risks.
Measure behavior Stress-test robustness, uncertainty, fairness, privacy leakage, abuse resistance, and failure modes.
Manage controls Apply mitigations, monitoring, oversight, incident response, documentation, and accountability.

23.1 Robustness and uncertainty

Learn:

  • Distribution shift: The deployment distribution differs from the training distribution.
  • Covariate shift: Input feature distributions change while the conditional label relationship is assumed stable.
  • Label shift: Class proportions change while class-conditional feature distributions are assumed stable.
  • Domain adaptation: Adapts models trained in one domain to perform reliably in another.
  • Out-of-distribution detection: Flags inputs that do not resemble training data or supported use cases.
  • Aleatoric uncertainty: Irreducible uncertainty from noise or ambiguity in the data-generating process.
  • Epistemic uncertainty: Reducible uncertainty from limited data, weak coverage, or uncertain model parameters.
  • Deep ensembles: Combine multiple independently trained models to improve uncertainty estimates.
  • Monte Carlo dropout: Samples predictions with dropout active at inference as an approximate uncertainty method.
  • Conformal prediction: Produces prediction sets with coverage guarantees under exchangeability assumptions.
  • Calibration: Checks whether predicted confidence matches observed accuracy.
  • Selective prediction: Allows the model to answer only when confidence or risk thresholds are satisfied.
  • Abstention: Refuses or escalates uncertain cases instead of forcing a low-confidence prediction.

23.2 Adversarial machine learning

Learn:

  • Evasion attacks: Modify inputs at inference time to induce incorrect behavior.
  • FGSM: Uses the sign of the input gradient to create a small adversarial perturbation.
  • PGD: Repeats projected gradient steps to search for stronger bounded perturbations.
  • Adversarial training: Trains on adversarial examples to improve robustness against known perturbation classes.
  • Data poisoning: Corrupts training data so the learned model behaves incorrectly.
  • Backdoors: Hide a trigger pattern that causes targeted behavior when present.
  • Model extraction: Attempts to reconstruct model behavior through queries or exposed outputs.
  • Membership inference: Infers whether a record was included in training data.
  • Model inversion: Recovers sensitive information about inputs or training data from model behavior.
  • Supply-chain attacks: Compromise models, datasets, packages, checkpoints, or infrastructure used in the AI pipeline.

FGSM:

x_adv = x + epsilon sign(grad_x L(theta, x, y))

23.3 Language-model and agent security

Study:

  • Prompt injection: Untrusted input tries to override the intended instruction hierarchy.
  • Indirect prompt injection: Malicious instructions arrive through retrieved documents, webpages, emails, files, or tool outputs.
  • Jailbreaks: Prompts attempt to bypass model safety constraints or policy boundaries.
  • Data exfiltration: Attempts to leak secrets, private data, hidden prompts, credentials, or protected context.
  • Tool manipulation: Steers an agent into unsafe or unintended tool calls.
  • Retrieval poisoning: Pollutes a retrieval index so the model grounds answers in attacker-controlled content.
  • Memory poisoning: Injects false or malicious state into long-term or session memory.
  • Output-parser attacks: Exploit fragile structured-output parsing or downstream command interpretation.
  • Unsafe code execution: Lets generated or retrieved code run without appropriate sandboxing and review.
  • Excessive agency: Grants a system too much autonomy, scope, or persistence for the risk level.
  • Privilege separation: Limits what each component, tool, credential, and runtime can access.
  • Capability-based security: Grants narrow, explicit capabilities instead of broad ambient authority.
  • Sandboxing: Runs untrusted actions in constrained environments with resource and permission limits.
  • Content provenance: Tracks where information came from and whether it should be trusted.
  • Red teaming: Systematically probes failures, abuse paths, and unsafe behavior before and after deployment.

23.4 Privacy

Learn:

  • Differential privacy: Bounds how much one record can affect observable outputs.
  • DP-SGD: Clips per-example gradients and adds noise during training to provide privacy guarantees.
  • Federated learning: Trains across distributed clients while keeping raw data local.
  • Federated averaging: Aggregates client updates into a shared model update.
  • Secure aggregation: Combines client updates without exposing individual updates to the server.
  • Homomorphic encryption concepts: Enables limited computation over encrypted data in privacy-preserving workflows.
  • Trusted execution environments: Run sensitive computation in hardware-backed isolated environments.
  • Data minimization: Collects, stores, and exposes only the data needed for a defined purpose.
  • Machine unlearning: Attempts to remove the influence of specific data from trained systems.

Differential privacy:

P[M(D) in S] <= exp(epsilon) P[M(D') in S] + delta

for neighbouring datasets D, D'.

23.5 Fairness and governance

Understand:

  • Demographic parity: Requires positive prediction rates to be similar across groups.
  • Equalized odds: Requires error rates such as true-positive and false-positive rates to be similar across groups.
  • Equal opportunity: Focuses on equal true-positive rates across groups.
  • Predictive parity: Requires predicted-positive cases to have similar precision across groups.
  • Calibration across groups: Checks whether confidence means the same thing for different groups.
  • Subgroup evaluation: Measures performance across meaningful slices instead of only aggregate metrics.
  • Dataset documentation: Records data provenance, collection limits, consent, labeling, coverage, and known risks.
  • Model cards: Summarize intended use, evaluation, limitations, risk controls, and deployment guidance.
  • Risk assessments: Identify severity, likelihood, mitigations, residual risk, and responsible owners.
  • Audit trails: Preserve decision records, model versions, data lineage, evaluations, and approval steps.
  • Human oversight: Defines when humans review, override, escalate, or stop automated behavior.
  • Incident response: Plans how to detect, triage, contain, disclose, and remediate failures.
  • Accountability: Assigns responsibility for design choices, approvals, monitoring, and harms.
  • Transparency: Communicates system purpose, limits, uncertainty, data use, and user-facing implications.
  • Explainability limits: Recognizes when explanations are incomplete, unstable, misleading, or not causal.

A practical risk workflow is to govern, map, measure, and manage AI risks across the full system lifecycle.

Why it matters for AI

AI systems fail through data shift, uncertainty, adversarial behavior, privacy leakage, unsafe tool use, unfair outcomes, and weak operational controls. Strong teams treat these as engineering requirements that must be measured, tested, mitigated, documented, and monitored.

Learning target

You should be able to explain robustness and uncertainty methods, adversarial ML threats, language-model and agent security risks, privacy-preserving training concepts, fairness metrics, documentation practices, oversight mechanisms, and a practical governance workflow for mapping, measuring, and managing AI risk.
