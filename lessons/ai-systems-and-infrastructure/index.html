24. AI Systems And Infrastructure

A modern AI practitioner must understand the system around the model: distributed training, efficient inference, deployment pipelines, monitoring, rollback, data and prompt versioning, and operational reliability.

System workflow

Train at scale Shard model state, data, activations, and communication across accelerator clusters.
Serve efficiently Optimize latency, throughput, memory, kernels, batching, caching, and cost per token.
Operate reliably Version artifacts, monitor behavior, evaluate continuously, collect traces, and recover from incidents.

24.1 Training systems

Study:

  • Data parallelism: Replicates the model across workers while each worker processes different batches.
  • Distributed data parallel: Synchronizes gradients across replicas, commonly through all-reduce communication.
  • Fully sharded data parallel: Shards parameters, gradients, and optimizer states across workers to reduce per-device memory.
  • ZeRO: Partitions optimizer states, gradients, and parameters across data-parallel ranks.
  • Tensor parallelism: Splits tensor operations and model layers across devices.
  • Pipeline parallelism: Splits model layers into stages and pipelines microbatches across devices.
  • Sequence parallelism: Splits sequence dimensions across devices to reduce memory or communication cost.
  • Context parallelism: Distributes long-context computation across devices or partitions.
  • Expert parallelism: Routes sparse mixture-of-experts computation across expert shards.
  • Gradient accumulation: Accumulates gradients over microbatches to simulate a larger batch size.
  • Activation checkpointing: Saves memory by recomputing selected activations during backpropagation.
  • Asynchronous data loading: Keeps accelerators fed by overlapping input loading, preprocessing, and host-to-device transfer.
  • Checkpoint sharding: Stores model checkpoints in shards suitable for distributed loading and recovery.
  • Fault tolerance: Recovers from worker failures, preemptions, corrupted checkpoints, or transient infrastructure issues.
  • Network collectives: Coordinate communication patterns such as reductions, gathers, and scatters across devices.
  • All-reduce: Aggregates values across workers and returns the result to every worker.
  • All-gather: Collects shards from all workers so each worker receives the full set.
  • Reduce-scatter: Reduces values and scatters shards of the reduced result across workers.
  • GPU topology: Accounts for device interconnects, nodes, links, and bandwidth when placing parallel work.
  • Compute and communication overlap: Hides communication latency by scheduling communication alongside useful computation.

Modern large-model courses explicitly include FLOPs, memory, arithmetic intensity, GPUs, custom kernels, distributed training, and scaling analysis.

24.2 Efficient inference

Learn:

  • Continuous batching: Dynamically packs requests into batches as they arrive and finish.
  • Dynamic batching: Groups requests within a short window to improve hardware utilization.
  • KV caching: Stores key-value tensors from prior tokens to avoid recomputing attention history.
  • Paged KV cache: Manages KV memory in blocks to reduce fragmentation and improve serving throughput.
  • Prefix caching: Reuses cached computation for shared prompt prefixes.
  • Quantization: Reduces numeric precision to lower memory bandwidth, storage, and compute cost.
  • GPTQ-style post-training quantization: Quantizes pretrained weights with calibration data and approximate second-order correction.
  • AWQ-style activation-aware quantization: Preserves important weight channels based on activation sensitivity.
  • SmoothQuant concepts: Smooths activation outliers into weights to make lower-precision inference easier.
  • Speculative decoding: Uses a smaller draft model and verifies tokens with a larger model to reduce latency.
  • Tensor parallel serving: Splits inference computation across multiple accelerators.
  • Pipeline serving: Serves model stages across devices, often trading latency for capacity.
  • Disaggregated prefill and decode: Separates prompt processing from token generation to optimize each phase independently.
  • Request scheduling: Chooses which requests run, batch, preempt, or resume under latency and memory constraints.
  • Memory offloading: Moves weights, caches, or intermediate state between device, host, and storage tiers.
  • Model compilation: Lowers model graphs into optimized kernels or execution plans.
  • Kernel fusion: Combines operations into fewer kernels to reduce memory traffic and launch overhead.
  • CUDA and Triton fundamentals: Provide the programming model behind many custom GPU kernels.

Key performance concepts

  • Throughput: Amount of work completed per unit time.
  • Time to first token: Delay before the first generated token is returned.
  • Inter-token latency: Delay between successive generated tokens.
  • Tail latency: Slow-end latency, often measured at p95, p99, or worse percentiles.
  • Memory bandwidth: Rate at which data moves between memory and compute units.
  • Arithmetic intensity: Ratio of compute operations to memory traffic.
  • Compute utilization: Fraction of available accelerator compute actually used.
  • Tokens per second: Generation or processing rate for token workloads.
  • Cost per million tokens: Serving cost normalized by token volume.
  • Energy use: Power and energy required to train or serve the system.

24.3 MLOps and LLMOps

Understand:

  • Data versioning: Tracks datasets, splits, preprocessing, labels, and lineage over time.
  • Experiment tracking: Records runs, metrics, hyperparameters, code versions, artifacts, and notes.
  • Model registries: Manage model versions, metadata, approvals, promotion, and deployment status.
  • Feature stores: Provide governed, reusable features for training and serving.
  • Reproducible pipelines: Make data processing, training, evaluation, and deployment repeatable.
  • Continuous integration: Tests model code, data transforms, pipelines, and system contracts before release.
  • Continuous deployment: Releases models and services through automated, controlled deployment paths.
  • Shadow deployment: Runs a candidate system beside production without serving user-visible results.
  • Canary deployment: Gradually sends a small traffic slice to a new model before broader rollout.
  • A/B testing: Compares variants under live or controlled traffic with defined metrics and guardrails.
  • Drift detection: Detects input, label, prediction, or performance distribution changes.
  • Monitoring: Tracks reliability, latency, cost, quality, safety, and data health in production.
  • Evaluation pipelines: Run repeatable offline and online evaluations for models, prompts, and retrieval systems.
  • Prompt versioning: Tracks prompt templates, system messages, tools, policies, and evaluation results.
  • Retrieval-index versioning: Versions embeddings, chunking, metadata, documents, and index configurations.
  • Trace collection: Captures request flows, tool calls, retrieval results, model outputs, and latency spans.
  • Feedback collection: Gathers human ratings, corrections, implicit signals, and failure reports.
  • Rollback: Returns to a known-good model, prompt, index, or service version after a failure.
  • Incident management: Defines detection, triage, ownership, mitigation, communication, and postmortems.

Why it matters for AI

Model quality depends on the surrounding system. Training infrastructure controls scale and reliability, inference infrastructure controls latency and cost, and operations infrastructure controls reproducibility, monitoring, deployment safety, and recovery.

Learning target

You should be able to explain major training parallelism strategies, key collective communication patterns, inference caching and batching methods, quantization and compilation concepts, performance metrics, deployment strategies, versioning needs, monitoring workflows, and incident-management practices for AI systems.
